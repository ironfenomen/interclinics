# Telegram-бот заявок «Интерклиника»: боевая модель

Документ фиксирует **финальную** структуру карточки, статусов, кнопок, полей, договорённостей, партнёрской ветки и отчётов. Реализация бота и хранилища должна опираться на **`lib/telegram-leads/model.ts`** (константы и типы) и этот файл (поведение и смыслы).

Текущий сайт шлёт уведомления через `POST /api/lead` (`phone`, `city`, `source`, `leadType`). Целевой контракт расширяется (см. §11); бот **не** живёт только в одном сообщении Telegram — состояние и история хранятся **в БД**, сообщение в чате — отображение + триггер действий.

---

## 1. Финальная структура Telegram-карточки новой заявки

Сообщение: **HTML** или **MarkdownV2** (единый стиль на проект), одна карточка на лид, далее **редактирование текста** того же `message_id` при смене статуса / договорённостей (или «якорное» сообщение + отдельные служебные — предпочтительно **одно сообщение на лид** с `editMessageText`).

### 1.1. Блок «событие и источник»

| Поле | Источник | Пример |
|------|-----------|--------|
| Тип события | константа | `Новая заявка` |
| Сайт | домен / `BRAND_DISPLAY_NAME` | `interclinics.ru` |
| Форма | `form_id` / human label | `hero`, `modal:callback`, `cta:stacionar-final` |
| Страница | path + query (без секретов) | `/stavropol/stacionar/?utm_source=yandex` |
| Устройство | UA-class | `mobile` / `desktop` / `tablet` |
| CTA-метка | внутренняя метка сценария | `hero:stacionar` |
| Referrer | `document.referrer` или заголовок | `https://yandex.ru/...` |
| UTM | склеенная строка | `utm_source=yandex&utm_medium=cpc&utm_campaign=...` |
| Время | ISO + МСК в тексте | `2026-04-06T14:32:01+03:00` (МСК) |
| Согласие на ПД | bool | `да` / `нет` |
| ID заявки | UUID или монотонный id из БД | `a1b2c3d4` |

### 1.2. Блок «контакт и запрос»

| Поле | Обяз. | Примечание |
|------|--------|------------|
| Имя | нет | из формы, если есть |
| Телефон | да | нормализованный E.164 / цифры в `<code>` |
| Город | да | slug + название |
| Тип услуги (лид) | да | маппинг `leadType`: выезд / стационар / реабилитация / общая |
| Что нужно / комментарий | нет | свободный текст, обрезка в карточке с «…» + полное в БД |

### 1.3. Блок «управление» (всегда внизу карточки, обновляется)

| Поле | Описание |
|------|-----------|
| Текущий статус | значение из `LEAD_STATUS` (человекочитаемо) |
| Ответственный | `@username` или имя из Telegram profile / «не назначен» |
| Результат | `LEAD_RESULT` кратко для оператора |
| Договорённости | последняя **открытая** строка + счётчик открытых |
| Партнёрский статус | `PARTNER_STATUS`, если применимо |
| Взято в работу | `taken_at` (МСК) |
| Без движения | если `updated_at` старше N часов при активном статусе — показать «⚠️ нет действий X ч» |

### 1.4. Пример текста карточки (шаблон)

```
🆕 <b>Новая заявка</b> · Интерклиника
🌐 Сайт: interclinics.ru
📋 Форма: <code>hero:stacionar</code>
📄 Страница: <code>/stavropol/?utm_source=...</code>
📱 Устройство: mobile
🏷 CTA: <code>hero:stacionar</code>
🔗 Referrer: …
📊 UTM: …
🕐 Время: 06.04.2026 14:32 МСК
✅ Согласие ПД: да
🆔 ID: <code>a1b2c3d4</code>

👤 Имя: Иван
📞 Телефон: <code>+7928…</code>
🏙 Город: Ставрополь · stavropol
📌 Тип: стационар
💬 Запрос: нужен стационар, созвонить после 18:00

────────────
<b>Управление</b>
Статус: Взято в работу
Ответственный: @operator
Результат: в работе
Договорённости: перезвон 07.04 11:00 (открыто: 1)
Партнёры: —
В работе с: 06.04 14:40
Движение: ок
```

## 2. Inline-кнопки под карточкой (основная ветка)

Ряд(и) кнопок под сообщением. Подписи — из `INLINE_BUTTON_LABEL` в `model.ts`.

| Действие (`INLINE_ACTION`) | Кнопка |
|----------------------------|--------|
| `take` | Взять в работу |
| `contacted` | Связались |
| `no_answer` | Не дозвонились |
| `agreements` | Договорённости |
| `callback_later` | Перезвонить позже |
| `awaiting_decision` | Ждём решения |
| `consultation` | Консультация |
| `booked` | Записан |
| `partner_sent` | Передано партнёрам |
| `refused` | Отказ |
| `spam` | Спам |

### 2.1. Правила отображения

- Кнопки **не** показываются все сразу вперемешку: сетка **2×N** или по смыслу группами (рабочие → исходы).
- В статусе `new` минимум: **Взять в работу**, **Спам**, **Отказ** (остальные можно скрыть до `take`).
- После **Передано партнёрам** основная воронка блокируется для «закрывающих» действий своей линии; показывается **партнёрский ряд** (§6).
- Любое нажатие: запись в **историю**, обновление БД, `editMessageText` + `editMessageReplyMarkup`.

### 2.2. `callback_data` (лимит 64 байта)

Рекомендуемый формат: `ic:{L}:{A}`

- `L` — короткий id заявки (например base36 от автоинкремента, 8–12 символов).
- `A` — значение из `INLINE_ACTION` или `PARTNER_INLINE_ACTION`.

Пример: `ic:9xK2m1:take` (проверять длину в UTF-8).

Для ввода договорённости второй шаг — **ForceReply** или **отдельное сообщение** с командой `/agree_{L}` — в callback не помещается длинный текст.

## 3. Финальный список статусов основной воронки

Соответствие константам `LEAD_STATUS`:

| Константа | Человекочитаемо |
|-----------|-----------------|
| `new` | Новая |
| `in_progress` | Взято в работу |
| `contacted` | Связались |
| `no_answer` | Не дозвонились |
| `agreements` | Договорённости |
| `callback_later` | Перезвонить позже |
| `awaiting_decision` | Ждём решения |
| `consultation` | Консультация |
| `booked` | Записан |
| `partner_sent` | Передано партнёрам |
| `refused` | Отказ |
| `spam` | Спам |
| `closed_won` | Закрыто: успех (своя линия) |
| `closed_lost` | Закрыто: не конвертировали |

**Смысл:** `booked` — оперативный «записан к нам»; `closed_won` можно выставить автоматически через N дней или вручную для финализации CRM. Минимально для отчётов достаточно `booked` как успех своей линии.

Переходы инициируются только кнопками (или админ-командами), с валидацией из допустимого графа (реализация).

## 4. Партнёрская ветка: статусы и кнопки

### 4.1. `PARTNER_STATUS`

| Константа | Кнопка / смысл |
|-----------|----------------|
| `none` | не у партнёров |
| `partner_in_progress` | Партнёры: в работе |
| `partner_agreements` | Партнёры: договорённости |
| `partner_passed` | Партнёры: прошла |
| `partner_not_passed` | Партнёры: не прошла |

### 4.2. Кнопки (`PARTNER_INLINE_ACTION`)

См. `PARTNER_INLINE_BUTTON_LABEL` в `model.ts`.

Показ: при `LEAD_STATUS.partner_sent` **или** когда партнёрская ветка уже тронута (не `none`), до финала `partner_passed` / `partner_not_passed`.

После `partner_passed` / `partner_not_passed` партнёрский ряд скрыть или оставить только «Открыть в CRM» (вне scope модели).

## 6. Договорённости

Сущность **Agreement** (много к одному лиду):

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | bigserial | |
| `lead_id` | fk | |
| `text` | text | текст договорённости |
| `created_by_tg_user_id` | bigint | кто внёс |
| `created_by_display` | string | |
| `created_at` | timestamptz | когда внёс |
| `next_contact_at` | timestamptz | обязательный следующий контакт |
| `record_status` | `AgreementRecordStatus` | `open` / `done` / `superseded` |

### 6.1. UX в боте

1. Нажатие **Договорённости** → бот просит ответом текст (ForceReply) или интерактив с датой (календарь — опционально): шаблон «текст + дата/время следующего контакта».
2. Парсинг → создание записи `open`, обновление `lead.status = agreements` (если ещё не), допись в `LeadActivity`, пересборка карточки.
3. Закрытие: кнопка «Выполнено» у последней договорённости или автоматически при переходе в `contacted` / `booked` — политика на реализацию; минимум — ручное «выполнено».

### 6.2. Использование в ежедневных сводках

Выборка: все лиды, у которых есть хотя бы одна запись `Agreement` с `record_status = open` **или** статус ∈ множеству из §7, с полем `next_contact_at` в горизонте (например сегодня±1 день) или просрочено.

## 7. Ежедневные сводки 10:00 и 18:00

- **Часовой пояс:** `Europe/Moscow`.
- **Канал:** тот же рабочий чат или отдельный «операторский» topic — настройка env.

### 7.1. Критерий попадания в сводку

Лид попадает, если выполняется **хотя бы одно**:

- `status` ∈ `DAILY_DIGEST_CORE_LEAD_STATUSES` ∪ `DAILY_DIGEST_EXTENDED_LEAD_STATUSES` (= `DAILY_DIGEST_LEAD_STATUSES` в `model.ts`), **и** `closed_at IS NULL`, **или**
- `partner_status` ∈ `DAILY_DIGEST_PARTNER_STATUSES` (`partner_in_progress`, `partner_agreements`), **и** итог партнёра ещё не финальный.

Ядро сводки по ТЗ: **Договорённости**, **Перезвонить позже**, **Ждём решения**; плюс партнёрские **в работе** / **договорённости**; расширение покрывает остальные активные follow-up (не дозвон, взято в работу, связались, консультация, передано партнёрам).

Дополнительно полезно включать лиды с **просроченным** `next_contact_at` у открытой договорённости независимо от статуса (реализация: отдельный запрос).

### 7.2. Строка сводки (для каждого лида)

| Колонка |
|---------|
| Имя |
| Телефон (маскирование опционально в чате) |
| Что нужно (обрезка) |
| Ответственный |
| Текущий статус (+ партнёрский если есть) |
| Текст последней открытой договорённости |
| Срок / дедлайн (`next_contact_at`) |
| `public_id` / ссылка на сообщение (если есть) |

Заголовок сообщения: `Договорённости и активные лиды — 10:00 06.04.2026 МСК` / аналогично для 18:00.

## 8. Еженедельная статистика

**Период:** календарная неделя пн 00:00 — вс 23:59:59 МСК (или вс–пн по политике компании — зафиксировать в env).

### 8.1. Показатели (ключи `WEEKLY_FUNNEL_KEYS`)

- `received_total` — новых лидов за неделю.
- `taken_in_progress` — сколько раз был переход в `in_progress` (или уникальные лиды — выбрать одно и держать).
- `contacted`, `no_answer`, `consultation`, `booked` — **события** перехода в этот статус за неделю (рекомендуется) **или** снимок — в отчёте явно написать метод.
- `partner_sent` — передано партнёрам за неделю.
- `refused`, `spam` — закрытия за неделю.
- `partner_in_progress`, `partner_agreements`, `partner_passed`, `partner_not_passed` — события/срез по партнёрам за неделю.

Источник истины — `LeadActivity`, агрегация SQL/BI.

---

## 9. Ежемесячная статистика

**Момент:** последний день месяца 23:59 МСК или 1-го числа 09:00 за прошлый месяц — зафиксировать в cron.

### 9.1. Блок «своя линия»

- `total_leads` — поступило за месяц.
- `passed_own` — `booked` + `closed_won` (определить правило).
- `not_passed_own` — `closed_lost` + часть `refused` (уточнить бизнес-правило).
- `in_progress_open` — не закрыты на конец месяца, не спам/отказ.
- `refused`, `spam`, `sent_to_partners`.

### 9.2. Блок «партнёры»

- `partner_passed`, `partner_not_passed`
- `partner_in_progress_open` — на конец месяца
- `partner_agreements_open` — открытые договорённости на партнёрской стороне

### 9.3. Доп. срезы (если позволяет модель)

Из полей лида: `owner_display`, `page_path` / `form_id`, `city_slug`, `lead_type` — сводные таблицы top-N в том же сообщении или вторым сообщением (лимит размера Telegram).

---

## 10. Потеря лида и восстановление

- Каждый вход с сайта: **INSERT Lead** + `LeadActivity` (`source:website`) до успешного `sendMessage`.
- Если `sendMessage` упал — ретрай с idempotency key = `public_id` из payload.
- Карточка всегда пересобирается из БД по команде `/lead {public_id}` для операторов (реализация).

---

## 11. Расширение контракта `POST /api/lead` (для полной карточки)

К текущим полям добавить (опционально, с дефолтами):

- `name`, `comment`, `formId`, `pagePath`, `pageQuery`, `deviceClass`, `ctaLabel`, `referrer`, `utm` (object или string), `consentPd`, `clientEventId` (idempotency).

Сервер нормализует и передаёт в сервис создания лида → Telegram.
